## What shipped (each slice = one green commit)

- S1 `src/core/tools.ts` — read-only context tools: grepRepo (ripgrep),
  readRange (exact line span, trailing-newline safe), gitContext.
- S2 `src/core/endpoint.ts` — local-first endpoint resolution (default local,
  cloud requires explicit key + is flagged), tolerant JSON extraction.
- S3 `src/core/author.ts` — LLM-sole-author skeleton. Injectable LLM; snippet
  text is re-fetched from disk and SHA-pinned (model's snippet is never trusted);
  tolerant-parse then regenerate-once; graceful skip when no endpoint.
- S4 — full 5-block prompt, per-plane Bloom targets + exemplars
  (`src/core/exemplars.ts`), teachability pre-filter (`src/core/teachability.ts`).
- S6a `src/core/verify.ts` — deterministic guardrails (format, anti-trivia,
  answer-leak) + commit-drift check (re-fetch pinned range at HEAD, compare).
- S6b `src/core/answerKey.ts` — independent answer-key validator. Blind derive
  (author's answer never enters the derivation prompt — asserted) then judge.
- S6c `src/core/grade.ts` + `src/core/staging.ts` — code-as-oracle grading
  (snippet is oracle, stored key is a reference) + pendingReview promotion gate.
- S5 `src/core/cardEval.ts` + `src/core/evalTargets.ts` — falsifiable eval:
  runs the REAL pipeline over held-out targets, scores (format .4 + answer-key
  .4 + freshness .2) vs a threshold (default .75). Seed targets cover all planes.
- S7a `src/core/graph.ts` — typed 2-edge graph (prereq_of + is_a), cycle
  detection, cycle-safe topo sort (throws on a prereq cycle).
- S7b `src/core/conceptState.ts` — ready/blocked/mastered; blocked is advisory,
  never a hard lock.
- S7c `src/core/scheduler.ts` — FSRS via ts-fsrs@5.4.1 (the one approved dep;
  exact-pinned, pure TS). Daily learning steps; Rating from verdict only (no
  confidence); defined mastery = retrievability@+7d discounted by graduation
  state and lapse ratio.
- S8a `src/core/safeStore.ts` — atomic write (temp+rename) + optimistic
  version check (StateConflictError on concurrent write). Non-destructive.
- S8b `src/core/budget.ts` — per-run ceilings (cards/tokens/optional cost),
  stops cleanly with a partial-but-valid run.
- S8c `skill/SKILL.md` — delivery wrapper. Every referenced CLI command verified
  real; encodes "the tutor is the oracle, don't substitute your own grep/LLM."
- S0 `src/core/featureFlags.ts` — v2 flag (MERGELEARN_V2, default OFF) +
  DEPRECATIONS registry (the authoritative S9 removal inventory). questions.ts
  gets an @deprecated banner, zero behavior change. cardBatches NOT deprecated.

## Design fidelity

The high-priority MoA/doc-05 fixes are all implemented and asserted by tests:
- Answer-key oracle problem → S6b blind derivation (blindness asserted) + S6c
  code-as-oracle grading (grader trusts snippet over a wrong stored key).
- Commit drift → S3 SHA-pinned frozen snippet + S6a drift detection.
- Confidence must not corrupt scheduling → S7c derives Rating from verdict only.
- Concurrent-writer corruption → S8a atomic + version-checked writes.
- Undefined mastery → S7c defined formula.
- Prereq hallucination → S7a topo sort throws on a cycle.

## Empirical findings (real local model, not mocks)

Ran the loop against a live Ollama model (qwen2.5-coder:1.5b, free/local) over
the 6 seed targets. Two findings:

1. The eval SCORER was flawed and is now fixed (commit "honest scoring"). The
   old weighted sum (format .4 + answerKey .4 + freshness .2) inflated results:
   freshness is ~always 1 by construction and format is cheap, so a card banked
   0.6 before any quality was measured. That produced a misleading 0.867. The
   scorer now treats authored/format/freshness as HARD GATES and drives the
   score off the one discriminating signal (independent answer-key agreement).

2. Under the honest scorer the same run scores 0.500 (< 0.75 threshold, FAIL),
   selfJudged=true. All 6 authored and passed structural gates (machinery is
   validated end-to-end), but answer-key agreement was 3/6 - and the judge was
   the same tiny model self-judging, so even that is weak evidence.

Conclusion: the LOOP is validated (author -> parse -> freeze -> SHA-pin ->
blind-derive -> judge -> score runs clean on real model output). Card QUALITY is
NOT validated. Reaching the bar needs the human-buffer work below (bigger model,
independent judge via the new judgeLlm seam, expanded held-out set, prompt
tuning) - none of which a one-shot run settles.

## Open decision (needs you)

S9 destructive migration is HELD. The design gates it on: (1) the author loop
validated empirically — run `runEval` against a real local LLM endpoint over the
seed targets and confirm score ≥ threshold; and (2) your explicit go-ahead,
because it rewrites `.skilltrace/state.json` and drops data-bearing collections
(see `dataBearingDeprecations()`). Nothing destructive has run.

Also inherently not one-shot-automatable and therefore NOT done here: empirical
prompt tuning against a live model, expanding exemplars/seed targets to the
~6–8/plane and ~20–30 the design calls for, and real-repo dogfooding. These need
a running local endpoint (Ollama/LM Studio) and human judgement.

## How to validate the loop (when you have a local endpoint)

1. Start a local OpenAI-compatible server (Ollama/LM Studio). Default base URL
   is already local; no cloud key is used.
2. Set MERGELEARN_V2=1 to opt into the v2 pipeline.
3. Call `runEval(repoPath, seedEvalTargets, { endpoint, commitSha })` and read
   `report.score` vs `report.threshold`. Tune the prompt/exemplars until it
   passes, then expand the seed set. This is the falsifiable "done" bar.
