## Customer and user analysis

The first ideal customer profile is a **solo or small-team AI-heavy developer working in a stack they do not fully know yet**, especially in TypeScript/JavaScript web codebases where AI-assisted development is already normalised and where the developer still owns merge, debugging, security, and maintenance outcomes. ## Market and competitor scan

The broader market signal is favourable. AI-assisted coding is no longer niche: Stack OverflowŌĆÖs 2025 survey reporting shows very high AI usage or planned usage among developers, while trust in correctness remains weak; Sonar-linked reporting in 2026 shows many developers using AI daily, relying on it for a large share of committed code, yet reviewing it inconsistently because validation feels costly. ## Evaluation plan

Before more substantial UI or optional LLM work, the platform needs a **real evaluation programme** that tests not just whether the output ŌĆ£looks smart,ŌĆØ but whether it produces useful, grounded, repeatable learning prompts. The current local evaluation harness is a good seed, but it is too synthetic and too small to de-risk the key questions. The next evaluation phase should therefore be built around a small corpus of real repositories, a human-scored rubric, and a short longitudinal user study. ŅłĆfileciteŅłéturn0file10Ņłü ŅłĆfileciteŅłéturn0file11Ņłü

The repo test set should include MergeLearn itself plus a deliberately varied set of TypeScript-heavy repositories: one Node CLI, one React app, one Next.js or similar full-stack app, one service/API backend, one library package, one repo with strong tests, one repo with meaningful docs, and one repo with high churn but modest architectural complexity. The point is not benchmark vanity. The point is to expose the extractor and planner to the kinds of code realities the first ICP will actually have: routes, schemas, tests, CLI flags, auth, typed APIs, and docs that may or may not match the code. That structure follows directly from the productŌĆÖs current TypeScript-first scope and from the need to improve beyond a single dogfood repo. ŅłĆfileciteŅłéturn0file6Ņłü ŅłĆfileciteŅłéturn0file8Ņłü

The minimum persona set should contain four user types. The first is the core ICP: an AI-heavy solo builder in a partly unfamiliar stack. The second is an experienced engineer using AI in a familiar repo, because that is where current productivity claims become most contested. The third is a more junior developer using AI to ramp into an unfamiliar codebase. The fourth is a repo expert or maintainer who can judge whether surfaced concepts and prompts are actually important. This follows from the mixed empirical picture in current AI-coding research: novices and peripheral contributors may gain more obvious throughput benefits, while experienced contributors often bear the integration and maintenance burden. ŅłĆciteŅłéturn20academia0Ņłéturn20academia5Ņłéturn2academia7Ņłü

The core study tasks should stay practical. Ask users to ingest recent history, run a first ŌĆ£TodayŌĆØ session, answer three cards, mark at least one bad card if needed, inspect ŌĆ£why this card appeared,ŌĆØ run ŌĆ£explain last commit,ŌĆØ and come back three to seven days later for delayed recall on one of the concepts. ŅłĆciteŅłéturn19academia1Ņłéturn21academia0Ņłéturn24academia1Ņłü

Metrics should be split into **content quality**, **behaviour**, and **learning effect**. For content quality, measure top-5 concept precision, evidence groundedness, specificity, answerability, duplication rate, and bad-card rate. For behaviour, measure time-to-first-useful-card, review completion rate, median session time, number of corrections made, and 7-day return. For learning effect, measure self-rated confidence before and after the first session, delayed recall on previously seen cards, and whether users can correctly trace or explain recent code changes in their own words. Retrieval-practice and explain-back research justify the delayed-recall emphasis; automated-feedback reviews in programming education justify tracking learner agency and higher-order learning rather than only correctness. ## Product roadmap

The next **one-week milestone** should focus on foundation rather than interface breadth. Concretely, that means: lock the evaluation rubric and fixture corpus, add deterministic card-quality gates, and add a minimal correction loop for bad cards and wrong concepts. If that week does not materially improve top-card precision and user trust, it is too early to expand the browser app further. ŅłĆfileciteŅłéturn0file11Ņłü

The next **one-month milestone** should then deliver the first truly sticky workflow: improved extraction, an interactive local ŌĆ£TodayŌĆØ session, and lightweight plan or focus-area scoping. At the end of that month, the product should be able to answer a very basic question convincingly: after a week of AI-heavy coding, do users voluntarily come back for three more cards because the cards are useful? That is the milestone that matters, not how many pages exist. ŅłĆfileciteŅłéturn0file4Ņłü ŅłĆfileciteŅłéturn0file8Ņłü

**Later and optional ideas** should follow only after those loops are proven.
