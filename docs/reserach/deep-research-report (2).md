# MergeLearn Tutor platform evaluation and next steps

## Executive verdict

Yes, this product direction is worth continuing, but only if it stays tightly focused on a problem that adjacent tools do not solve well: **helping AI-heavy developers convert recent code changes into durable understanding**. The strongest version is not a generic “AI coding tutor”, not a PR blocker, and not a repo chat wrapper. It is a **local-first, repo-aware, evidence-linked micro-tutor** that uses recent commits, diffs, PRs, and user corrections to identify “understanding debt” and turn it into one short, repeatable review habit. That framing fits both the current MVP and the best-supported opportunity in the research. AI-assisted coding is now widespread, agent-authored PRs are already visible at large scale, and multiple studies show that productivity gains are mixed while verification, trust calibration, and code comprehension remain unresolved human bottlenecks. citeturn15academia0turn3academia4turn12academia12turn11academia3turn12news0

“Knowledge debt” is a good internal framing and a reasonably strong product framing, but it should not be the only external message. In research and industry discussion, the closest observed pains are low trust in AI output, rising verification effort, weaker understanding of generated code, and the risk of skill erosion or overreliance. For messaging, the better front-door promise is closer to **“understand what you shipped while using AI”** or **“stay fast with AI without losing the plot”**. “Knowledge debt” works well as the mental model behind the product and in the dashboard/debt views, but user-facing copy should lead with faster re-understanding, safer AI use, and steady skill growth. citeturn5news0turn11news0turn15academia5turn14academia5turn0file1

The strongest version of the product is therefore: a **personal learning companion for AI-heavy TypeScript/JavaScript developers** that stays local by default, extracts grounded concepts from recent repo activity, asks for short explain-back responses, schedules lightweight review, and lets users correct the system when it guesses wrong. That direction matches the current MVP’s local-first CLI, transparent state, deterministic extraction, daily cards, and static dashboard, but the platform is still too shallow and too evaluation-light to justify major UI expansion. The next phase should be about **evidence and calibration**, not feature breadth. fileciteturn0file1 fileciteturn0file3 fileciteturn0file4

The main things to avoid are building a generic course platform, a manager/surveillance dashboard, mandatory cloud sync, default LLM grading, a heavy IDE extension too early, or a “perfect” knowledge graph before concept quality and repeatable habit value are proven. Research on automated feedback and AI tutors repeatedly shows that many systems over-index on automation, correctness, and short-term novelty while under-serving higher-order learning, learner agency, and long-term outcomes. MergeLearn Tutor should not repeat that pattern. citeturn2academia0turn2academia1turn10academia6turn10academia5

## Customer and user analysis

The best first ICP is **solo or small-team AI-heavy developers using Cursor/Copilot/Claude Code on a TypeScript or JavaScript repo that they do not fully understand yet**. This segment has strong pain, low procurement friction, high tolerance for CLI/local tooling, and enough repo ownership to dogfood a local-first workflow. It is stronger than “junior developers” as a first wedge because juniors often need broader pedagogy, institutional support, and manager buy-in, while solo and startup engineers can adopt immediately and judge value on their own work. The current MVP is already tailored to that audience through `today`, `review`, `debt`, `map`, and `explain-last-commit`, plus a transparent local learner state. citeturn12academia12turn3academia3turn3academia6turn12news0 fileciteturn0file1 fileciteturn0file5

The core job-to-be-done is not “teach me programming” in the abstract. It is: **“Help me understand the code and concepts I just shipped with AI, while gradually getting better at this stack and this repo.”** That job appears in several adjacent research lines. Studies of GenAI in code comprehension argue that programmers increasingly need to understand generated solutions to verify and integrate them. Studies of trust in Copilot-like tools show that programmers need trust calibration rather than blind confidence. Course-aware tutor deployments also report that learners primarily want conceptual understanding, implementation guidance, and debugging help, not unrestricted answer generation. citeturn15academia5turn11academia3turn10academia6turn14academia7

Existing alternatives are insufficient because each solves only one slice. AI coding assistants help generate or edit code, but not durable learning. AI code review tools help inspect diffs, but not build a persistent learner model. Repo documentation/search tools help explain repositories, but not infer what *you* are weak on. Onboarding tools and code tours help newcomers, but are usually static or expert-authored rather than adaptive to your actual commit history. General programming education tools teach languages and frameworks, but not “TypeScript as used in this repo”. That gap is exactly where MergeLearn Tutor can be differentiated. citeturn7view6turn9view0turn16view0turn17view2turn13view0turn7view0turn10academia5

The smallest weekly habit loop should be **two short, context-triggered moments**, not a full “study routine”. First, a **post-work recap** after a meaningful coding session or merge: “What did I just touch, what matters, and what should I be able to explain?” Second, a **weekly debt review** of 10–15 minutes: revisit 3–5 important concepts from recent work, answer 2–3 explain-back prompts, correct any wrong concept tagging, and pick one “next focus” topic. That is more realistic than trying to drive a daily five-minute habit from day one. Research on supportive programming interventions suggests structured, supportive designs outperform punitive ones, and course-aware AI tutors are used most for short, situated help rather than constant engagement. citeturn2academia3turn10academia6turn19academia2

If there is ever a buyer beyond the end-user, the most plausible future buyer is not a manager buying surveillance dashboards. It is an **onboarding lead, staff engineer, or small engineering org** that wants repo-specific learning scaffolding for new joiners or AI-heavy teams. The LACY results are especially important here: expert-guided code tours beat AI-only tours on quiz outcomes, and users preferred tours over self-study. That supports a later team/onboarding motion, but it should remain a later expansion, not the default product shape now. citeturn10academia5

## Market and competitor scan

The market context is strong enough to justify continued work. AI coding is no longer niche: a large-scale 2026 GitHub study estimated coding-agent adoption across projects at roughly **15.85%–22.60%**, and the AIDev dataset collected **932,791 agent-authored PRs** across **116,211 repositories**. At the same time, reported benefits are uneven. A 2026 survey-and-literature synthesis found that developers report the biggest GenAI benefits in implementation, testing, and documentation, while planning and requirements benefit less. A controlled METR study found experienced OSS developers were **19% slower** on familiar codebases with AI tools, even though they expected to be faster. That combination — widespread adoption plus mixed real productivity — is precisely why tools focused on *understanding and review of one’s own AI-assisted work* are strategically interesting. citeturn3academia4turn15academia0turn12academia12turn12news0

Pain points are also consistent across sources. Developers use AI heavily, but trust remains low. Secondary reporting on the 2025 Stack Overflow survey says usage rose while distrust in AI outputs increased, and Sonar’s 2026 survey reporting says most developers do not fully trust AI-generated code, yet many still do not review every AI-authored change because verification is time-consuming. Research on trust evolution suggests the right intervention is **trust calibration**: programmers still need comprehension, debugging, and testing skills even when trust rises through experience. MergeLearn Tutor fits that gap far better than another generation-first assistant. citeturn5news0turn11news0turn11academia3

In the AI coding assistant category, tools like GitHub Copilot, Cursor, Claude Code, Amp, Cody, Devin, and Tabnine help users generate, edit, debug, or plan code. Sourcegraph’s Cody explicitly positions itself as an assistant that uses development context to help developers understand, write, and fix code faster, and it pulls context from local and remote codebases. This is adjacent, but these products still optimise for helping the current coding task, not for maintaining a persistent user knowledge model or scheduling learning from repo history. citeturn17view2turn16view2turn10search0

In AI code review and governance, GitHub Copilot code review, CodeRabbit, and Qodo are the most relevant comparators. GitHub documents Copilot code review as reviewing code and providing feedback with suggested changes. CodeRabbit markets itself around cutting review time and bugs, finding defects, summarising diffs, and reviewing PRs, IDE work, and CLI work. Qodo positions itself as code review and governance with full-codebase intelligence, rules, PR history, and cross-repo context. These products are close in “context-aware review” but still fundamentally answer **“what is wrong with the code?”** or **“does the change meet standards?”** rather than **“what does this developer still need to understand?”** citeturn8view5turn9view0turn16view0

In codebase understanding and documentation, Sourcegraph Code Search, Cody, DeepWiki, and Swimm are relevant. Sourcegraph emphasises finding, understanding, and fixing code through code search and navigation. DeepWiki describes itself as “up-to-date documentation you can talk to, for every repo in the world.” Swimm has now shifted far from lightweight documentation into an “agentic context layer” and validated knowledge base for AI tools and migrations. These categories prove there is demand for repo understanding and context layers, but they stop short of turning repo history into a personalised, longitudinal learning system. citeturn17view0turn17view2turn13view0turn9view2

In onboarding and code tours, CodeTour and LACY are especially relevant. CodeTour is explicitly about recording and replaying guided walkthroughs of a codebase for onboarding, re-boarding, bug reports, and PR context. LACY extends that concept with expert-curated and AI-generated tours, quizzes, and reusable onboarding support; the expert-guided version materially outperformed AI-only tours. This is the best adjacent evidence that MergeLearn Tutor should eventually support **curated repo narratives**, but for the personal MVP it implies that AI-generated lessons alone are not enough: users need editability, pinning, and correction. citeturn7view0turn10academia5

In programming education and spaced repetition, tools like Exercism, Hyperskill, Anki, and Mnemosyne show that programming learners will use structured projects, instant feedback, and spaced recall systems. But those tools are curriculum-first, not repo-first. Educational research also supports retrieval practice, self-explanation, and spacing, while warning that automated feedback systems often remain too correctness-focused and too weak on long-term learning. MergeLearn Tutor’s opportunity is to **apply those learning mechanisms to real repository work**, not to out-compete mainstream learning platforms on course content. citeturn10search2turn10search8turn2search4turn19academia2turn19academia1

The upshot is that the market is crowded around generation, review, documentation, and onboarding, but still relatively open around **repo-aware personal tutoring**. That makes the category promising, but only if MergeLearn Tutor stays disciplined and does not collapse into “yet another assistant that explains code”. citeturn15academia5turn10academia5turn16view0turn17view2

## Product critique of current MVP

The current MVP has a **good directional spine**. It is local-first, TypeScript/Node-based, reads git commits and diffs, stores transparent state in `.skilltrace/state.json`, generates daily learning cards, records explain-back answers, renders a profile/debt report/Mermaid graph/static dashboard, avoids telemetry, avoids executing target repo code, and does not require an LLM. Those are strong trust-building choices, especially when contrasted with cloud assistants that collect prompts, responses, and usage data. fileciteturn0file1 fileciteturn0file4 citeturn17view2

The CLI is coherent and already expresses the intended product loop well. The command surface includes `init`, `ingest`, `today`, `review`, `profile`, `debt`, `map`, `explain-last-commit`, `dashboard`, and `answer`, which is a sensible control plane for a local-first tutor. The problem is not that the CLI is wrong; it is that the CLI and static dashboard currently make the experience largely **read-only and report-like**, whereas a tutoring product must earn repeat use through an interactive review session. fileciteturn0file1 fileciteturn0file5

The static dashboard is useful as an inspection surface, but weak as a habit surface. It renders “Today”, debt, cards, a ledger table, and a Mermaid graph into a static HTML file, which is enough to show the architecture, but not enough to support session completion, answering, skipping, marking ambiguity, fixing wrong concepts, or tracking why a concept was prioritised. This matters because the current learning science evidence strongly favours interactive recall, self-explanation, and feedback over passive reading. A static dashboard is a fine report; it is not yet a tutor. fileciteturn0file5 citeturn19academia1turn19academia2turn19academia0

Concept extraction is the most important weak point. The current extractor uses a small deterministic concept catalogue with regex patterns and path hints for concepts like type aliases, interfaces, union types, generics, async/await, React hooks, behaviour tests, auth boundaries, validation, CLI tools, and dependency management. It also creates “repo domain” concepts by splitting changed file paths into tokens and turning those into concepts. That is a reasonable baseline, but still shallow: it risks false positives from path names, misses symbol-level structure, and likely collapses distinct repo ideas into noisy labels. Before more UI is built, this part needs better evaluation and a correction loop. fileciteturn0file5

The mastery model is directionally improved from the dogfood critique — passive exposure is capped at 35% until active recall happens — but it is still heuristic and not yet credible enough for strong product claims. The planner sets default mastery from exposure count, importance from concept kind/evidence volume, and priority from repo relevance, importance, weakness, and due time. When answers are recorded, mastery moves up or down by fixed deltas. That is acceptable for an MVP, but there are two major issues. First, answer correctness is currently marked manually through the CLI `--correct` flag, which means the system does not actually evaluate answers or capture uncertainty; it simply records the user’s supplied outcome. Second, the event model in types is richer than the actual interaction surface: the schema allows `shown`, `skipped`, `marked_wrong`, and `marked_correct`, but the recorded command path only creates `answered` events. In practice, this means the learner model is still under-instrumented. fileciteturn0file3 fileciteturn0file5

The “knowledge debt” framing is strong, but the current MVP risks turning debt into a **reporting metaphor** rather than a **behaviour-changing loop**. The current artefacts do a good job telling the user what the system thinks is weak, but they do a weaker job helping the user clear that debt in one consistent weekly behaviour. The platform should move from “dashboard + cards” towards “one short session that always ends in either recall, correction, or deferral”. That would also make the debt framing feel less abstract and more actionable. fileciteturn0file1 fileciteturn0file3

The local-first stance and the lack of LLM by default are both major strengths. They keep installation and trust simple, avoid unnecessary cloud exposure, differentiate the product from cloud-first assistants, and align with the stated product stance. This is particularly valuable because several adjacent assistants explicitly collect prompts/responses or usage data in order to provide their services. That said, local-first needs to become more than a stance: it should become a tested boundary with outbound-preview UX, ignore paths, redaction policies, and explicit logs if LLM experiments are later introduced. fileciteturn0file1 fileciteturn0file4 citeturn17view2

JSON versus SQLite is the least urgent debate right now. The repo explicitly chose JSON to keep Node 20 setup simple and dependency-free, and the store is already isolated behind a store module. That was the right MVP trade-off. SQLite will become justified when the product needs richer query patterns, more events, correction histories, session states, and faster local UI reads. At the present stage, the bigger risk is not JSON itself; it is that the product might build more UI before validating concept quality, correction behaviour, and repeatable session value. fileciteturn0file1 fileciteturn0file4

## Evaluation plan

Before major new implementation, the platform needs a **three-layer evaluation program**: concept extraction quality, learning-session usefulness, and habit/return value. Research on automated feedback in programming education shows that evaluations often stop at short-term performance or satisfaction and miss long-term learning, learner agency, and instructional fit. MergeLearn Tutor should not expand the UI until it proves that its concept tags and cards are both **grounded** and **worth coming back to**. citeturn19academia2turn10academia6turn10academia5

The repo test set should include at least eight TypeScript/JavaScript repositories across four shapes: a small utility/library repo, a medium web app, a backend/API service, and the user’s own repo. At least half should be real public repos with active tests and recent changes, and at least two should be repos the test users are already familiar with. This matters because AI tool performance and perceived value differ on familiar versus unfamiliar codebases; the METR result is a warning that “AI help” can look very different once users know the repo well. citeturn12news0turn12academia12

The user test should include three cohorts: AI-heavy solo/startup developers, developers joining or re-entering an unfamiliar repo area, and a small number of junior-to-mid developers using AI heavily on TypeScript. Each participant should run the same lightweight script: initialise the tutor on a real repo, ingest 30–60 days of history, complete one “today” session, answer three prompts, inspect the debt view, correct any wrong concepts, and return after three to seven days for a follow-up review. The goal is not to prove educational superiority yet; it is to find out whether the product is *grounded, trusted, and reusable*. citeturn10academia6turn10academia5turn11academia3

The quantitative metrics should focus on the parts that most directly determine whether this should become a platform rather than remain a clever demo. For extraction quality, measure concept precision at top 3 and top 5, evidence correctness, duplicate/noise rate, and percentage of concepts users actively correct. For card usefulness, ask raters to judge each card on relevance, clarity, answerability, actionability, and whether it taught them something they did not already confidently know. For habit value, track whether users voluntarily return within a week, whether they complete at least one full session, and whether they report that the session helped them understand or re-understand recent work faster than their usual approach. These measures map better to the actual product thesis than generic quiz scores alone. citeturn5academia9turn19academia1turn10academia6

The qualitative interview questions should be strict and concrete. Ask: “Which card was clearly wrong?” “Which card made you notice something important?” “What did the product think you knew that you did not?” “Which concept labels felt too generic?” “When would you realistically run this: after coding, at the end of the week, during onboarding, or never?” “Would you trust this in a private repo?” “What is the minimum number of cards you would tolerate per week?” These questions directly test the current risks identified by the research packet: generic extraction, false mastery, weak habit surfaces, and privacy/trust. fileciteturn0file3 fileciteturn0file6

The failure criteria should be explicit. Stop major UI buildout if any of the following happen in the first real-user round: concept precision at top-3 useful concepts is below 70%; more than 20% of cards are judged irrelevant or unanswerable; fewer than 40% of users voluntarily return for a second session within a week; median session time exceeds eight minutes for a “today” review; users cannot explain why a mastery/confidence score changed; or users express clear discomfort about privacy boundaries. Those are the kinds of failures that no amount of dashboard polish will fix. citeturn19academia2turn2academia1

Privacy and safety checks should be evaluated as first-class acceptance criteria, not buried in documentation. The tests should confirm that the tool makes no network calls by default, runs no target repo code, respects ignore paths, and stores all state locally. If an experimental LLM mode is added later, the evaluation must also check whether the tool previews exact outbound content, records what left the machine, supports redaction, and allows a local-model path. The privacy boundary is part of the product, not an implementation detail. fileciteturn0file1 fileciteturn0file4 citeturn17view2

## Recommended next platform steps

The next implementation work should be deliberately narrow and dependency-ordered. The first five steps should improve *truthfulness and reuse* before they improve aesthetics.

**Task one is a multi-repo evaluation harness and scoring rubric.**
**Goal:** establish whether concept extraction and card generation are good enough to justify more product investment.
**Why now:** current local evaluation only checks that a synthetic repo produces a few expected concepts and cards; it does not test usefulness, precision, or user correction needs across real repos. fileciteturn0file5
**Files or modules likely affected:** `src/eval/localEvaluation.ts`, new evaluation fixtures, CLI/reporting docs, possibly `src/core/render.ts` for machine-readable evaluation output.
**Acceptance criteria:** at least eight repos can be evaluated reproducibly; each run emits a structured report covering concept precision, evidence correctness, and card usefulness ratings; the report can be compared across runs.
**Tests and evaluation required:** golden evaluation outputs on fixture repos; one manual rating rubric document; one automated smoke run in CI.
**User-visible outcome:** credibility that the system is not just demoing itself on an easy fixture.
**Risk if skipped:** the team will optimise UI around a weak extractor and misread output prettiness as value.

**Task two is a user correction loop for concepts and cards.**
**Goal:** let users mark concepts as wrong, merge duplicates, down-rank generic tags, and save repo-specific names or explanations.
**Why now:** LACY and adjacent onboarding work strongly suggest expert curation matters, and the current extractor is too heuristic to remain uneditable. citeturn10academia5
**Files or modules likely affected:** `src/core/types.ts`, `src/core/planner.ts`, `src/core/store.ts`, `src/cli.ts`, the dashboard layer, and likely new correction-specific modules.
**Acceptance criteria:** a user can mark a concept/card as wrong; can attach a better label or explanation; the same mistake is suppressed or reduced in future sessions; corrections are visible in local state.
**Tests and evaluation required:** integration tests showing that a correction changes future prioritisation and rendering; regression tests for persistence.
**User-visible outcome:** the product feels collaborative rather than patronising.
**Risk if skipped:** trust will collapse as soon as the system guesses wrong on a real repo.

**Task three is an AST-backed TypeScript concept extractor with evidence ranking.**
**Goal:** improve concept precision and evidence quality beyond regex-and-path heuristics.
**Why now:** concept quality is the biggest technical bottleneck, and TypeScript is the chosen first language.
**Files or modules likely affected:** `src/core/concepts.ts`, `src/core/conceptCatalog.ts`, likely a new AST analysis module, and possibly `src/core/git.ts` for snippet extraction.
**Acceptance criteria:** extraction supports symbol-aware detections for common TypeScript patterns such as exported functions, interfaces, unions, generics, async flows, tests, routes, validation schemas, and imports/usages; evidence is attached at file/symbol/hunk level rather than mostly whole-file level.
**Tests and evaluation required:** fixture coverage for false positives and true positives; measurable precision improvement over regex baseline on the evaluation harness.
**User-visible outcome:** fewer generic or obviously wrong cards.
**Risk if skipped:** adding richer review UX will simply magnify extraction noise.

**Task four is a richer learner-event model and session state.**
**Goal:** capture `shown`, `answered`, `skipped`, `unsure`, `hinted`, `corrected`, and `deferred` events, plus basic session boundaries.
**Why now:** the current learner model cannot tell the difference between “user ignored this”, “user was uncertain”, and “user confidently answered”, yet these differences are essential if the product is meant to model understanding. fileciteturn0file5
**Files or modules likely affected:** `src/core/types.ts`, `src/core/planner.ts`, `src/core/store.ts`, `src/cli.ts`, the eventual local web review UI.
**Acceptance criteria:** every review session generates structured events; concept state changes are explainable from those events; the product can show why a concept is due and why mastery/confidence changed.
**Tests and evaluation required:** state transition tests; session replay tests; transparent audit rendering in profile/debt outputs.
**User-visible outcome:** the learner ledger starts to feel intelligible and trustworthy.
**Risk if skipped:** mastery remains opaque and fragile.

**Task five is an interactive local review surface.**
**Goal:** replace the static dashboard as the primary habit surface with a local web session that supports answering, skipping, marking uncertainty, and correcting content.
**Why now:** current UX is too report-like; the next surface should make the weekly loop frictionless.
**Files or modules likely affected:** `src/dashboard/html.ts` or a replacement local web app, plus planner/store/session modules.
**Acceptance criteria:** a user can open a local session, complete 3–5 items, record actions without leaving the UI, and end with an updated debt/profile summary.
**Tests and evaluation required:** end-to-end session smoke tests; usability test with real users completing a session in under eight minutes.
**User-visible outcome:** the product becomes something users can *do*, not just something they can read.
**Risk if skipped:** repeat use will remain weak, because the main visible surface is static.

**Task six is an editable concept-pack and repo-lexicon layer.**
**Goal:** let users or future teams define concept packs, repo terms, prerequisites, and “important flows” without changing code.
**Why now:** the product needs a path from generic TypeScript concepts to “TypeScript as used in this repo”, and editability is more important than ontology perfection at this stage.
**Files or modules likely affected:** concept catalogue, store, planner, CLI, docs.
**Acceptance criteria:** concept definitions can be extended with local YAML/JSON files; repo-specific aliases and pinned concepts appear in extraction and review.
**Tests and evaluation required:** fixture repo with custom concepts; precedence and merge tests.
**User-visible outcome:** the product starts to feel repo-aware rather than merely language-aware.
**Risk if skipped:** the knowledge graph remains generic and less defensible.

**Task seven is a privacy boundary and outbound-preview layer for optional enrichment.**
**Goal:** prepare for future LLM experiments without compromising the product’s local-first trust model.
**Why now:** optional LLM enrichment is probably useful later, but only if the privacy UX exists first.
**Files or modules likely affected:** new provider abstraction, config layer, store/event logs, UI copy, docs.
**Acceptance criteria:** no remote model can be used without explicit configuration; the user sees exactly what content would be sent; local-model support is possible; outbound calls are logged locally.
**Tests and evaluation required:** network-off default tests; configuration tests; redaction tests.
**User-visible outcome:** trust is maintained even when experimentation begins.
**Risk if skipped:** LLM features will feel like a betrayal of the product’s core stance.

**Task eight is a narrow opt-in LLM enrichment experiment.**
**Goal:** test whether LLMs improve explanation quality and card phrasing *without* becoming the truth source.
**Why now:** after evaluation, corrections, stronger extraction, and privacy boundaries, LLM enrichment can be tested safely on a narrow axis.
**Files or modules likely affected:** provider abstraction, planner/card generation, UI surfacing.
**Acceptance criteria:** the experiment is limited to rewriting explanations, generating worked examples, or offering alternate phrasings; no default grading; every enriched card remains evidence-linked; user can compare deterministic vs enriched variants.
**Tests and evaluation required:** A/B rating of card usefulness on a fixed sample; privacy boundary verification.
**User-visible outcome:** higher-quality explanations where it truly helps.
**Risk if skipped:** content may remain dry or awkward; but this is a lower risk than shipping ungrounded AI too early.

## Product roadmap

The **next one-week milestone** should be brutally practical: ship the evaluation harness design, the correction model design, and enough instrumentation planning to make the next real-user round meaningful. In other words, spend the week making the current MVP falsifiable. The deliverable should be a multi-repo evaluation protocol, a first correction UX plan, and a clear definition of how the learner event model changes when users answer, skip, or correct content. That week should end with a go/no-go decision on whether concept quality is strong enough to justify an interactive local review UI. fileciteturn0file6 fileciteturn0file7

The **next one-month milestone** should be a “truthful tutor beta”, not a flashy platform. By the end of that month, the product should support multi-repo evaluation, AST-backed TypeScript extraction, user corrections, a basic interactive local review surface, and a more transparent learner event model. It is acceptable if the graph remains simple and the visual polish remains modest; what matters is that users can run real sessions, correct bad guesses, and come back for a second round. That is the point at which packaging, naming, licensing, and a wider OSS release become worth serious attention. fileciteturn0file1 fileciteturn0file3

**Later and optional ideas** include a read-only MCP/IDE surface, GitHub PR ingestion, repo-specific code tours, curated “important flow” lessons, local-model or enterprise-model enrichment, and a future onboarding/team mode based on curated tours and shared concept packs. Those are promising precisely because the research suggests expert guidance and contextual tutoring beat generic AI-only onboarding, but they should follow proof of the personal weekly habit, not precede it. citeturn10academia5turn10academia6

The things **explicitly not to build yet** are a manager dashboard, organisation-level surveillance, mandatory cloud sync, always-on telemetry, LLM grading as the truth source, multi-language support beyond a carefully chosen next language, heavy gamification, certificate/course mechanics, and a full IDE plugin. Those all increase complexity and policy surface area before the core evidence loop is proven. ## Technical architecture recommendations

On storage, the right decision today is **not** “migrate immediately to SQLite” or “never migrate”. It is: keep the current JSON store for the next validation slice, but design the next features so that a future storage swap is easy. Once corrections, session histories, local interactive queries, and multiple review passes become routine, SQLite will become the cleaner default. For now, JSON remains defensible because the repo intentionally prioritised Node 20 reliability and store isolation over early database complexity. fileciteturn0file1 fileciteturn0file4

On concept graph representation, the product should remain pragmatic. Use stable concept IDs, concept packs, evidence references, and a simple layered graph: language/framework concepts, repo-domain concepts, and user evidence. Do not overbuild prerequisite logic yet. Knowledge-space style DAG thinking is useful, but the first architectural priority is to support **correction, aliasing, and pinned repo concepts**. A slightly messy but editable graph is better than a theoretically elegant graph users cannot fix. citeturn15academia1turn10academia5

On the LLM boundary, the system should stay **local-first and deterministic-first**, with an explicit adapter seam for opt-in enrichment. The model should never be the default truth source for extraction or grading. Its later role should be constrained to explanation rewriting, worked-example generation, alternate card wording, or summarising already-grounded repo facts. Every outbound call should be previewable; every sent artefact should be logged locally; ignore-paths and redaction should be first-class. This is the right privacy/UX response when many adjacent assistants are cloud-mediated and collect prompts or usage data. citeturn17view2 fileciteturn0file4

On surfaces, the next step should be **local web-first for the habit loop, CLI-first for control, IDE-later for convenience**. The current CLI should remain the ingestion and power-user surface. The next habit surface should be an interactive local review session, because that best matches actual repeat use. IDE integration should wait until the session and extraction quality are proven, otherwise the product will become another interruptive sidebar without a compelling weekly routine.
